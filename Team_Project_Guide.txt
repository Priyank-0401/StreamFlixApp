==============================================================================================================================
                              SUBSCRIPTION BILLING & REVENUE MANAGEMENT SYSTEM — TEAM PROJECT GUIDE
==============================================================================================================================

The system is structured into independent domains aligned with a real-world subscription billing architecture
to enable parallel development, ownership, and scalability.

This document is the single source of truth for our team of 5. It covers:
  1. What the project is (in plain English)
  2. System Integration Flow (how domains connect)
  3. The 5 domains — what each one covers
  4. Detailed task breakdown per domain
  5. All 15 User Stories (9 from SRS + 6 new)
  6. Tech stack per domain
  7. Database schema (provided in SRS)
  8. API endpoints (provided in SRS)

Read everything carefully. Each person will own ONE domain end-to-end (Backend + Database for that slice). No overlapping.

==============================================================================================================================
                                                   SECTION 1 — PROJECT OVERVIEW
==============================================================================================================================

Think of this project as the billing engine that powers companies like Netflix, Spotify, or any SaaS product.

When a user signs up, the system doesn't just charge them once. It has to:
  - Remember to charge them every month (or year)
  - Handle what happens if their credit card fails
  - Calculate taxes based on their region
  - Apply promo codes and coupons
  - Automatically adjust charges if they upgrade or downgrade mid-cycle
  - Send reminders before billing
  - Show finance teams how much revenue the business is generating

Here is what the system does from start to finish:

  1. CATALOGING     — Admins create Plans (e.g., Basic, Pro) and set prices.
  2. ONBOARDING     — Customers create accounts, pick a plan, enter payment details (mocked, not real).
  3. THE ENGINE     — A background timer (Cron Job) wakes up daily, checks who needs to be billed, calculates their total (taxes, discounts), and generates an invoice.
  4. COLLECTIONS    — It attempts to charge the card. If it succeeds, the customer gets a receipt. If it fails, the system retries later (Dunning) and eventually suspends the account.
  5. CUSTOMER CARE  — Support agents can log in to issue refunds or pause accounts if a customer complains.
  6. REPORTING      — Finance managers see dashboards showing revenue (MRR/ARR) and cancellations (Churn).

ROLES IN THE SYSTEM (Authentication & RBAC):
  - Customer        → subscribes, manages payment methods, views invoices
  - Admin           → manages products, plans, price books, coupons, taxes
  - Finance Manager → invoices, collections, revenue reports
  - Support Agent   → assists customers, refunds/adjustments

==============================================================================================================================
                                             SECTION 2 — SYSTEM INTEGRATION FLOW
==============================================================================================================================

This is how the 5 domains connect to each other in the live system:

    Customer ──→ [Auth & Product Mgmt] ──→ [Subscription Mgmt] ──→ [Billing & Invoicing] ──→ [Payments & Pricing] ──→ [Analytics & Notifications]
                                                                            │
                                                                            ▼
                                                                    [Analytics & Notifications]

  Data flows left to right:
    1. Customer authenticates and browses the product catalog                                       (Domain 1)
    2. Customer subscribes to a plan; system manages lifecycle (trial, active, pause, cancel)       (Domain 2)
    3. Billing engine detects renewals, generates invoices, and schedules retries on failure         (Domain 3)
    4. Payment service executes the actual charge, applies taxes/coupons, processes refunds          (Domain 4)
    5. Analytics computes revenue metrics; Notifications sends receipts, reminders, and alerts       (Domain 5)

  CRITICAL BOUNDARY CLARIFICATION (to avoid confusion during integration):
    - Billing & Invoicing (Domain 3) decides WHEN to retry a failed payment (the schedule: day 1, day 3, day 7).
    - Payments & Pricing (Domain 4) executes the actual payment attempt when called by Billing.
    - Domain 3 owns the retry schedule and dunning state machine. Domain 4 owns the payment gateway interaction.

  ENTITY STATE OWNERSHIP (which domain is the source of truth for each core entity):
    - Customer, Product, Plan           → Domain 1 (Auth & Product Management)
    - Subscription, SubscriptionItem    → Domain 2 (Subscription Management)
    - Invoice, InvoiceLineItem, Credit  → Domain 3 (Billing & Invoicing)
    - Payment, TaxRate, Coupon          → Domain 4 (Payments & Pricing)
    - Notification, AuditLog, Metrics   → Domain 5 (Analytics & Notifications)

  FAILURE PROPAGATION FLOW:
    All failures propagate upward through the system:
    Payment failure (Domain 4) → Billing marks past_due & schedules retry (Domain 3) → Subscription status updated (Domain 2) → Customer notified (Domain 5)

  IMPLEMENTATION APPROACH:
    The system is logically designed as microservices to maintain clean domain boundaries. However, for practical development
    efficiency, it will be implemented as a modular monolith (single codebase with clear module separation). Each domain
    maintains its own package/module, own database tables, and communicates through well-defined internal interfaces.
    This reduces deployment complexity while preserving architectural integrity.

------------------------------------------------------------------------------------------------------------------------------
DOMAIN 1: AUTH & PRODUCT MANAGEMENT [Login, Roles, Catalog]
------------------------------------------------------------------------------------------------------------------------------

Backend Microservices: auth-service, customer-service, catalog-service, api-gateway

What This Domain Covers:

  AUTHENTICATION & AUTHORIZATION
    - Set up Spring Security with OAuth2 (JWT tokens)
    - Implement Role-Based Access Control (RBAC) for 4 roles: Admin, Finance Manager, Support Agent, Customer
    - Build login (POST /auth/login) and token refresh (POST /auth/refresh)
    - Ensure unauthorized access returns 401/403 errors
    - API Gateway routing and entry-point configuration

  CUSTOMER MANAGEMENT
    - Customer registration and profile management
    - APIs: POST /customers, GET /customers/{id}, PUT /customers/{id}, DELETE /customers/{id}, GET /customers?email=&status=
    - Store: name, email, phone, currency, address, status

  PRODUCT CATALOG
    - Admin CRUD for Products, Plans, Add-ons, Metered components
    - Price books per currency with effective start/end dates
    - Plan attributes: billing interval, trial days, setup fee, tax behavior, proration behavior
    - APIs: POST /products, GET /products, POST /plans, GET /plans?productId=&currency=, POST /addons, GET /addons

Frontend Pages:
    - Login / Signup screens (all roles)
    - Admin Console → Product, Plan, Add-on management screens
    - Customer Portal → Profile management section

Database Tables Owned:
    - Customer
    - Product
    - Plan
    - (Add-on table if needed)

Tech Stack:
    - Spring Boot, Spring Security (OAuth2), Spring Cloud Gateway
    - Spring Data JPA, MySQL
    - Angular or React (Frontend)
    - Swagger/OpenAPI for API docs

------------------------------------------------------------------------------------------------------------------------------
DOMAIN 2: SUBSCRIPTION MANAGEMENT [The Subscription Lifecycle]
------------------------------------------------------------------------------------------------------------------------------

Backend Microservices: subscription-service

What This Domain Covers:

  SUBSCRIPTION LIFECYCLE
    - Full state machine: draft → trialing → active → past_due → paused → canceled
    - Create subscription (link customer + plan + payment method)
    - Start trials with configurable trial_days from the Plan
    - Activate subscription on trial end or immediate start
    - Cancel subscription (immediately or at period end)
    - Pause subscription with pause_from and pause_to dates
    - Resume paused subscriptions (revert to active)
    - Support multiple subscriptions per customer

  MID-CYCLE CHANGES (PRORATION)
    - Upgrade or downgrade a plan mid-billing cycle
    - Calculate proration based on remaining days in current period
    - Option to charge immediately or defer to next renewal
    - Proration lines must appear on the resulting invoice
    - Update subscription items; record changes in audit log

  SUBSCRIPTION ITEMS
    - Each subscription can have multiple items (base plan, add-ons)
    - Track item_type, unit_price, quantity, tax_mode, discount_id

  APIs:
    - POST /subscriptions
    - GET /subscriptions/{id}
    - PATCH /subscriptions/{id}                                  (for upgrades, downgrades, pause, resume)
    - POST /subscriptions/{id}/cancel?at_period_end=true/false
    - GET /customers/{id}/subscriptions

Frontend Pages:
    - Customer Portal → Subscription Onboarding Wizard (Choose plan, add-ons, billing cycle)
    - Customer Portal → Manage Plan (Upgrade / Downgrade / Pause / Cancel buttons)
    - Support Console → View customer subscription history & take actions

Database Tables Owned:
    - Subscription
    - SubscriptionItem

Tech Stack:
    - Spring Boot, Spring Data JPA, MySQL
    - Angular or React (Frontend)
    - Swagger/OpenAPI for API docs

------------------------------------------------------------------------------------------------------------------------------
DOMAIN 3: BILLING & INVOICING [Scheduled Billing, Invoices, Dunning Retries]
------------------------------------------------------------------------------------------------------------------------------

Backend Microservices: billing-service, invoice-service

What This Domain Covers:

  BILLING ENGINE (THE CRON JOB)
    - Build the core Scheduler using Spring @Scheduled
    - Cron detects subscriptions due for renewal in the next 24 hours
    - Trigger invoice generation for each due subscription
    - On-demand billing trigger via API: POST /billing/run
    - Track billing jobs: GET /billing/jobs?status=
    - Idempotency: ensure no duplicate invoices for the same period

  INVOICE GENERATION
    - Generate invoice with: header, line items, taxes, discounts, totals
    - Invoice statuses: draft → finalized → paid → void
    - Line items include description, quantity, unit price, amount, tax_rate_id, discount_id
    - Support for Credit Notes linked to invoices
    - PDF generation for invoice download
    - APIs: GET /invoices?customerId=&status=, GET /invoices/{id}, POST /invoices/{id}/finalize, POST /invoices/{id}/void, POST /invoices/{id}/pay

  DUNNING — RETRY SCHEDULING (this domain decides WHEN to retry; Domain 4 executes the payment)
    - On payment failure: mark subscription as past_due
    - Define and manage retry schedule: e.g., retry after 1 day, 3 days, 7 days
    - Trigger card-update reminder notification after each failure (calls Domain 5's notification API)
    - After max retries: cancel subscription or put on_hold (configurable policy)
    - Track retry attempt count, schedule, and dunning state machine

  ⚠️ BOUNDARY NOTE: This domain decides WHEN to retry. It calls Domain 4's payment API to execute the actual charge.

Frontend Pages:
    - Customer Portal → Invoice list with status, view/download as PDF
    - Finance Dashboard → Invoices aging view, collections status
    - Admin Console → Billing jobs dashboard (job runs, statuses)

Database Tables Owned:
    - Invoice
    - InvoiceLineItem
    - CreditNote
    - (BillingJob / DunningSchedule tables as needed)

Tech Stack:
    - Spring Boot, Spring @Scheduled (Cron), Spring Data JPA, MySQL
    - PDF library (e.g., iText or Apache PDFBox)
    - Angular or React (Frontend)
    - Swagger/OpenAPI for API docs

------------------------------------------------------------------------------------------------------------------------------
DOMAIN 4: PAYMENTS & PRICING [Payment Processing, Taxes, Coupons, Refunds]
------------------------------------------------------------------------------------------------------------------------------

Backend Microservices: payment-service

What This Domain Covers:

  MOCK PAYMENT GATEWAY
    - Simulate payment processing (no real gateway)
    - Payment tokenization: store tokens, never store raw card/PAN numbers
    - Idempotent charge endpoint: use idempotency keys to prevent duplicate charges
    - Simulate webhook responses (success/failure)
    - Record every payment attempt with gateway_ref, amount, status, attempt_no, response_code

  PAYMENT PROCESSING
    - APIs: POST /payments, GET /payments/{id}
    - Link payments to invoices
    - Handle payment statuses: pending → succeeded / failed

  ⚠️ BOUNDARY NOTE: This domain EXECUTES payment attempts. Domain 3 (Billing) decides WHEN to call this service for retries.

  REFUNDS
    - Full and partial refund with a reason field
    - API: POST /payments/{id}/refund
    - Auto-create a Credit Note when refund is issued
    - Update revenue reports on refund

  TAX ENGINE
    - Region-based tax rules: GST, VAT
    - Support inclusive and exclusive tax modes
    - Tax rate effective date ranges
    - APIs: POST /tax-rates, GET /tax-rates
    - Rounding policy and minor-unit (paise/cents) storage

  COUPON ENGINE
    - Coupon types: percentage or fixed amount
    - Duration: once, repeating (N months), forever
    - Expiry dates and max redemption limits
    - Validate coupon applicability
    - APIs: POST /coupons, GET /coupons, POST /coupons/validate

Frontend Pages:
    - Customer Portal → Payment method management, Checkout payment form
    - Admin Console → Tax rules configuration, Coupon creation/management
    - Support Console → Issue Refund button, Credit Note view

Database Tables Owned:
    - Payment
    - TaxRate
    - Coupon

Tech Stack:
    - Spring Boot, Spring Data JPA, MySQL
    - Angular or React (Frontend)
    - Swagger/OpenAPI for API docs

------------------------------------------------------------------------------------------------------------------------------
DOMAIN 5: ANALYTICS & NOTIFICATIONS [Dashboards, Reports, Alerts]
------------------------------------------------------------------------------------------------------------------------------

Backend Microservices: reporting-service, notification-service

This domain handles 3 distinct concerns, each with a different data access pattern:
    - Analytics    → read-heavy (aggregations, snapshots, dashboards)
    - Notifications → event-driven (triggered by billing/subscription events)
    - Audit Logs   → append-only (immutable, compliance-focused)

What This Domain Covers:

  REVENUE ANALYTICS
    - Calculate KPIs:
        MRR  (Monthly Recurring Revenue)
        ARR  (Annual Recurring Revenue)
        ARPU (Average Revenue Per User)
        Gross Churn & Net Churn rates
        LTV  (Lifetime Value — basic calculation)
        DSO  (Days Sales Outstanding)
    - Daily snapshots of metrics
    - Filters: by time range, plan, region
    - CSV and PDF export of reports
    - APIs: GET /reports/revenue?from=&to=&granularity=, GET /reports/mrr?from=&to=, GET /reports/churn?from=&to=

  NOTIFICATION SERVICE
    - Templates for: renewal reminders, receipts, payment failures, upcoming invoice alerts
    - Mock Email (SMTP) and Mock SMS sending
    - Renewal reminders at T-7 days and T-1 day before billing (configurable intervals)
    - Respect customer notification preferences (opt-in/opt-out)
    - Mandatory transactional notifications cannot be opted out of
    - Maintain notification logs (who was sent what, when, status)

  AUDIT LOGGING
    - Capture all monetary actions system-wide
    - Log fields: Actor, Action, Entity Type, Entity ID, Old Value, New Value, Request ID, IP, Timestamp
    - Immutable: logs cannot be edited or deleted from frontend
    - Filterable by Customer ID, Action Type, Date Range

Frontend Pages:
    - Finance Dashboard → Charts & Graphs for MRR, ARR, Churn, ARPU → Export center (CSV/PDF download buttons)
    - Customer Portal → Notification preferences (toggle switches)
    - Admin Console → Audit log viewer with filters

Database Tables Owned:
    - Notification
    - AuditLog
    - (RevenueSnapshot / MetricDaily tables as needed)

Tech Stack:
    - Spring Boot, Spring Data JPA, MySQL
    - Angular or React (Frontend)
    - Chart.js or Recharts (for dashboard graphs)
    - Swagger/OpenAPI for API docs

                                               SECTION 7 — DATABASE SCHEMA (FROM SRS)
==============================================================================================================================

All amounts are stored in minor units (paise for INR, cents for USD). Currency follows ISO-4217. Time is stored in UTC.

  Customer(id, name, email, phone, currency, address, status, created_at, updated_at)

  Product(id, name, description, status, created_at, updated_at)

  Plan(id, product_id, name, billing_period, price_minor, currency, trial_days, setup_fee_minor, tax_mode, effective_from, effective_to, status)

  Subscription(id, customer_id, plan_id, status, start_date, current_period_start, current_period_end, cancel_at_period_end, paused_from, paused_to, payment_method_id, currency)

  SubscriptionItem(id, subscription_id, item_type, unit_price_minor, quantity, tax_mode, discount_id)

  Invoice(id, customer_id, subscription_id, status, issue_date, due_date, subtotal_minor, tax_minor, discount_minor, total_minor, balance_minor, currency)

  InvoiceLineItem(id, invoice_id, description, quantity, unit_price_minor, amount_minor, tax_rate_id, discount_id, metadata)

  Payment(id, invoice_id, gateway_ref, amount_minor, currency, status, attempt_no, response_code, created_at)

  TaxRate(id, region, rate_percent, inclusive, effective_from, effective_to)

  Coupon(id, code, type, amount, duration, duration_in_months, max_redemptions, redeemed_count, valid_from, valid_to, status)

  CreditNote(id, invoice_id, reason, amount_minor, created_by, created_at)

  AuditLog(id, actor, action, entity_type, entity_id, old_value, new_value, request_id, ip, created_at)

  Notification(id, customer_id, type, subject, channel, status, sent_at)

==============================================================================================================================
                                            SECTION 10 — KEY WORKFLOWS (FROM SRS)
==============================================================================================================================

  WORKFLOW 1: SUBSCRIPTION CREATION
    Customer selects plan & payment method → System validates coupon/tax (Domain 4) → Creates subscription (Domain 2)
    → If setup fee or immediate charge: generate invoice (Domain 3) → attempt payment (Domain 4)
    → On success: send receipt (Domain 5) → On failure: mark past_due, enqueue retry (Domain 3)

  WORKFLOW 2: RENEWAL BILLING (DAILY CRON)
    Scheduler finds subscriptions expiring next day (Domain 3) → Generates invoice with proration/discounts/taxes (Domain 3)
    → Attempts payment via Domain 4's API (idempotent) → Updates subscription status (Domain 2) → Sends receipt or failure notice (Domain 5)

  WORKFLOW 3: UPGRADE/DOWNGRADE MID-CYCLE
    Customer requests plan change (Domain 2) → System computes proration credit/debit (Domain 2)
    → Charge now or defer to next renewal → Updates subscription items → Records change in audit log (Domain 5)

  WORKFLOW 4: DUNNING & RETRY
    Payment fails (Domain 4 reports failure) → Domain 3 marks past_due and schedules retries
    → Domain 3 calls Domain 4 to execute each retry → Domain 5 notifies customer to update payment method
    → After max attempts: cancel or put on_hold per policy (Domain 3)

==============================================================================================================================
                                                SECTION 11 — IMPORTANT NOTES
==============================================================================================================================

  1. All monetary amounts are stored in MINOR UNITS (paise, not rupees). Example: ₹499.00 is stored as 49900.
  2. All internal timestamps are in UTC. Frontend displays in the user's local timezone.
  3. Payment gateway is MOCKED — no real payment processing. Email/SMS is also MOCKED — no real sending.
  4. Each domain owner is responsible for writing unit tests for their microservices (target ≥ 80% coverage).
  5. Every domain owner must provide Swagger/OpenAPI documentation for their APIs.
  6. Use Bean Validation (@Valid, @NotNull, etc.) for all API inputs.
  7. Use a standard error response format across all services: { timestamp, requestId, errorCode, message, details, path }
  8. Centralized exception handling via @ControllerAdvice in each service.
  9. Structured JSON logs with correlationId and requestId in every service.
  10. IDEMPOTENCY: Idempotency ensures that duplicate billing attempts (due to retries or network failures) do not result in
      multiple charges for the same invoice. Every payment and invoice creation request must include an idempotency key.
      If the same key is received twice, the system returns the original response without re-processing.
  11. Frontend development will be centralized under one team member to maintain UI/UX consistency across all portals
      (Customer Portal, Admin Console, Finance Dashboard, Support Console). All other domain owners provide API contracts
      and test their endpoints via Postman/Swagger.

==============================================================================================================================
                                       SECTION 12 — INTEGRATION & COLLABORATION RULES
==============================================================================================================================

  To avoid blockers and merge conflicts, follow these rules:

  1. API-FIRST DEVELOPMENT
     - Each domain owner defines their API contracts BEFORE writing code.
     - For every API endpoint, provide:
         a) Request JSON schema (with sample payload)
         b) Response JSON schema (success + error)
         c) Error cases and HTTP status codes
     - Share Swagger/OpenAPI specs so frontend and other domains can develop in parallel.
     - No domain should start backend coding until its API contract is reviewed and agreed upon by the team.

  2. NO CROSS-DOMAIN DATABASE ACCESS
     - Never directly query another domain's tables. Always use REST APIs to fetch data from other domains.
     - Example: Billing (Domain 3) calls GET /subscriptions/{id} to get subscription details — it does NOT query the Subscription table directly.

  3. COMMUNICATION PROTOCOL
     - Each domain exposes REST APIs. Inter-service calls are synchronous REST over HTTP.
     - For notifications (receipts, reminders), Domain 3 calls Domain 5's notification API.

  4. WHO CALLS WHOM (dependency map):
     - Domain 2 (Subscriptions) calls → Domain 1 (to validate customer/plan) and Domain 4 (to validate coupon)
     - Domain 3 (Billing) calls → Domain 2 (to get subscriptions due), Domain 4 (to execute payment), Domain 5 (to send notifications)
     - Domain 4 (Payments) calls → no outbound calls (it is called by Domain 3)
     - Domain 5 (Analytics) calls → Domain 2, Domain 3 (to aggregate data for reports)
